Reference¶

versionutils.versioning¶

register(cls[, manager_name='versions'[, changes_tracker=None]]):

Registers the model class cls as a versioned model. After registration (and a call to syncdb) changes to the model will be tracked.

Args:

cls: The class to be versioned.

manager_name: Optional name of the attribute that’s added to
instances of cls. This is set to versions by default.
changes_tracker: An optional instance of ChangesTracker.
See the documentation on extending for more information.

Versioning manager¶

Versioned models have a versioning manager associated with them. Unless you’ve provided a special manager_name when using register(), this manager is named versions.

You can access the manager on both instances and model classes. The manager has all the standard QuerySet methods like all() and filter() plus some methods and shortcuts specific to versioning.

Example:

# Display all versions of MyModel
MyModel.versions.all()

# Display all versions of instance of MyModel
m = MyModel.objects.get(name="Foo")
m.versions.all()

Please see the usage documentation.

In addition to the standard QuerySet methods, we also provide:

most_recent()¶

This method must be called via an instance of the model rather than from the class. E.g. m.history.most_recent() rather than MyModel.history.most_recent().

Returns:
The most recent historical record instance.
Raises:
DoesNotExist: Instance has no historical record.
as_of([date=None][, version=None])¶

This method must be called via an instance of the model rather than from the class. E.g. m.history.as_of(..) rather than MyModel.history.as_of(..).

Args:
date: A datetime object. The datetime doesn’t have to be
exact. We will return the historical instance that’s most recent, moving backward in time.

version: An integer version number.

Returns:
A historical record instance that represents the object as it was at the date or version number provided.
Raises:
DoesNotExist: Instance hasn’t been created yet.

Historical instances¶

Methods on the versioned manager produce either historical instances or QuerySets of historical instances.

A historical instance represents a model instance at some point in time. A historical model contains all the same fields, attributes and methods as the usual model, but also has some extra methods and version metadata.

Here are the extra methods and version metadata that we provide on historial instances:

revert_to([delete_newer_versions=False][, **kws]):

Reverts to this version of the model. Example:

>> ph = p.versions.as_of(..)
>> ph.revert_to()
Args:
delete_newer_versions: If True, delete all versions in the
history newer than this version.
track_changes: If False, won’t log this revert as an action in
the history log.
kws: Any other keyword arguments you want to pass along to the
model save method.
version_info¶

Versioning metadata for the the historical instance. You can easily define new fields to track here.

Built-in attributes are:

date¶

DateTimeField representing the momement in time the historical instance references.

version_number()¶

Returns the version number of the historical instance.

type_to_verbose()¶

Returns a human-readable description of the type of action the historical instance refers to. E.g. “Updated”

Optional fields, on by default (see how to extend):

user¶

Optional UserField. Automatically set if you have AutoTrackUserInfoMiddleware enabled (see how to install)

user_ip¶

Optional IPAddressField. Automatically set if you have AutoTrackUserInfoMiddleware enabled (see how to install)

comment¶

Optional CharField intended to be a user’s description of the change action.

Fields you probably don’t need to use:

type¶

An integer representing the type of action the historical instance refers to. You should probably use type_to_verbose().

id¶

A global versioning id for the instance. This is global for the whole versioned model’s table. Use version_number(), it’s almost certainly what you want.

versionutils.versioning.forms¶

class versionutils.versioning.forms.CommentMixin(*args, **kwargs)¶

ModelForm mixin that adds a comment field to the form. When the ModelForm is saved the comment is set on the historical instance.

NOTE: You need to list CommentMixin before ModelForm in your subclasses list. Example:

class PageForm(CommentMixin, forms.ModelForm):
    class Meta:
        model = Page
        fields = ('content',)

versionutils.versioning.views¶

A set of class-based views that make working with versioned models a little more generic. These views are modeled after django’s class-based generic views.

class versionutils.versioning.views.UpdateView(**kwargs)¶

A subclass of django.views.generic.UpdateView that that displays a form for editing an existing object, setting a message if the form was successfully merged, and redisplaying the form with validation errors (if there are any) and saving changes to the object. This uses a form automatically generated from the object’s model class (unless a form class is manually specified).

You can use this with or without versioning.forms.MergeMixin forms.

success_msg()¶

Returns a string that gets sent to django.contrib.messages‘ add_message() when the model is successfully saved.

class versionutils.versioning.views.DeleteView(**kwargs)¶

A subclass of django.views.generic.DeleteView. A view that displays a confirmation page with a comment form and deletes an existing object, passing along the comment to the historical instance. The given object will only be deleted if the request method is POST. If this view is fetched via GET, it will display a confirmation page.

The delete confirmation page displayed to a GET request uses a template_name_suffix of ‘_confirm_delete’.

success_msg()¶

Returns a string that gets sent to django.contrib.messages‘ add_message() when the model is successfully deleted.

template_name_suffix¶

By default this is _confirm_delete.

form_class¶

By default this is DeleteForm, which can be found in versioning.forms.

class versionutils.versioning.views.RevertView(*args, **kwargs)¶

A view for reverting an object retrieved with self.get_object, with a response rendered by template. We display a confirmation page with a comment form and deletes an existing object, passing along the comment to the historical instance. The given object will only be deleted if the request method is POST. If this view is fetched via GET, it will display a confirmation page.

The delete confirmation page displayed to a GET request uses a template_name_suffix of ‘_confirm_revert’.

template_name_suffix¶

By default this is _confirm_revert.

form_class¶

By default this is RevertForm, which can be found in versioning.forms.

class versionutils.versioning.views.VersionsList¶

A subclass of django.views.generic.ListView.

get_queryset()¶

You need to provide this method. It should return a queryset containing historical instances. For instance:

def get_queryset(self):
    all_page_versions = Page(slug=self.kwargs['slug']).versions.all()
    # We set self.page to the most recent historical instance of the
    # page.
    self.page = all_page_versions[0]
    return all_page_versions
context_object_name¶

By default this is versions_list.

template_name_suffix¶

By default this is _versions.

revert_view_name¶

By default we look for a view with the name app_label:revert. If you provide a string for revert_view_name we’ll use that as our revert view.
